# PokerGame PRD

## 文档目的

本文定义 PokerGame 当前原型必须保留和后续改动必须尊重的用户可见行为。它面向继续实现或调整玩法、UI、AI 难度的开发者。

本文仅根据当前仓库可见内容整理；未在仓库中找到证据的需求均标记为 `待确认`，不作为已确认范围。

## 适用范围

适用：

- Godot 4 + GDScript 本地 Texas Hold'em 原型。
- 玩家与 1-5 个 AI 对手的单人牌局。
- 菜单、牌桌、玩家行动、AI 行动、摊牌结果和下一手牌流程。
- 简单、中等、困难三档 AI 的用户可见区别。
- 中文界面、深夜像素美术、事件日志、设置弹窗和本地聚合战绩。

不适用：

- 真实货币、账号、联网对局、排行榜、云存档、Steamworks、移动端发布、外部 API 或 LLM 对手。
- 当前仓库没有证据的商业化、剧情或多人同步需求。

## 用户和使用场景

| 用户 | 场景 |
| --- | --- |
| 本地玩家 | 打开 Godot 项目，选择 AI 数量和难度，进行单人 Texas Hold'em 对局。 |
| 维护者 | 修改规则、AI 或 UI 后，通过测试 runner 和人工烟测确认没有破坏核心玩法。 |

## 功能需求

| 编号 | 需求 |
| --- | --- |
| PRD-1 | 玩家必须能从菜单选择 1-5 个 AI 对手。 |
| PRD-2 | 玩家必须能选择 Simple、Medium、Hard 三档难度。 |
| PRD-3 | 开始游戏后必须创建一名人类玩家和所选数量 AI，发两张手牌并进入 preflop。 |
| PRD-4 | 每手牌必须处理按钮位、大小盲、preflop/flop/turn/river/showdown 阶段。 |
| PRD-5 | 玩家在自己回合只能执行当前合法行动：fold、check、call、raise、all-in。 |
| PRD-6 | 加注只有在合法时出现，并且金额必须受最小加注和最大可投入筹码限制。 |
| PRD-7 | AI 行动必须通过规则引擎校验，不得直接修改筹码、下注或阶段。 |
| PRD-8 | 无人跟注时必须直接结算底池；多人进入摊牌时必须比较最佳五张牌。 |
| PRD-9 | all-in 情况必须支持边池；平局必须支持分池。 |
| PRD-10 | 手牌结束后必须显示获胜者、赢得金额和牌型，并提供下一手或重开入口。 |
| PRD-11 | Hard AI 座位必须显示已分配个性标签，且其决策来源包含 preflop 起手牌评分、postflop Monte Carlo、底池赔率和个性参数。 |
| PRD-12 | 用户可见界面和结果默认使用中文；牌型与事件记录不能只在 HUD 层局部翻译。 |
| PRD-13 | AI 行动必须分时执行：简单/中等使用 3-5 秒随机等待，困难根据人格使用不同随机范围；等待时间只影响呈现，不影响决策强度。 |
| PRD-14 | 设置弹窗必须保存 AI 数量、难度、音效和音乐开关，并显示总手数、胜手数、净盈利和最大单手收益。 |
| PRD-15 | 统计重置必须两次确认，第一次点击后设置弹窗保持打开并在原位置显示确认状态。 |
| PRD-16 | 牌桌必须显示最近真实事件，不用占位行补齐日志；大小盲标识必须在后续行动后保持可见。 |
| PRD-17 | 菜单、牌桌、角色、卡牌和行动控件应遵守 `docs/art-direction.md`；动态中文、牌值、花色和金额必须由运行时生成。 |

## 非功能需求

| 类别 | 要求 |
| --- | --- |
| 本地性 | v1 保持离线运行，不加入网络请求、账号、遥测、LLM 或外部服务。 |
| 技术栈 | 使用 Godot 4 + GDScript；除非用户明确要求，不引入 C#、第三方插件或外部扑克库。 |
| 可维护性 | 游戏规则、AI 决策和 UI 必须分层，减少 UI 改动影响规则正确性的风险。 |
| 可验证性 | 行为改动必须优先用 `tests/test_runner.gd`；UI/资源改动还必须运行 `tests/ui_layout_probe.gd` 和主场景启动检查。 |
| 可移植性 | 当前窗口配置为 1280x720、canvas_items stretch、expand aspect；移动端/Steam 适配仍待确认。 |
| 视觉一致性 | 使用七色核心色板、硬边像素与 nearest-neighbor 过滤；不把动态文字和游戏数值固化进 PNG。 |

## 边界情况

- 玩家面对下注时不能 check。
- 玩家或 AI 筹码不足时，call 或 blind 只能支付剩余筹码并进入 all-in。
- 只剩一个未弃牌玩家时，必须跳过后续街道并结算无人争夺底池。
- 多名 all-in 玩家必须按投入额构造边池。
- 公共牌形成共享最大牌时，必须允许分池。
- 若一局后少于两名玩家有筹码，当前代码会重置筹码开始新手牌；该行为属于当前实现证据，但是否符合长期产品设计仍待确认。

## 非目标

- 不实现真钱或筹码购买。
- 不实现联网、多设备同步或在线匹配。
- 不实现账号、玩家资料、排行榜、云保存。
- 不实现 Steam API、成就、云存档或内购。
- 不引入 LLM 对手、外部服务、遥测或后台数据收集。
- 不把当前字体样张视为可发布字体，也不承诺当前 UI 已完成移动端适配。
- 不保存完整手牌历史、当前对局或云端资料。

## 实现指引

- 新玩法需求必须先判断属于规则层、AI 层还是 UI 层，再落到对应目录。
- 任何用户行动或 AI 行动都必须调用 `PokerRound.apply_action()`。
- 任何摊牌比较都必须调用 `HandEvaluator.evaluate()` 或 `HandEvaluator.compare_results()`。
- UI 可以读取状态并展示合法行动，但不应直接改写 `players` 的筹码、下注、状态或阶段。
- AI 可以计算策略，但返回的行动仍必须由规则引擎验证。
- 本地设置和聚合统计通过 `LocalProfile` 读写，不得成为另一份牌局权威状态。
- 新增或替换美术资源前先阅读 `docs/art-direction.md`，并以 `main.gd` 的 `preload()` 判断实际运行时依赖。

## 验收标准

- 从菜单能分别以 1、3、5 个 AI 开局。
- 三档难度都能完成至少一手牌。
- 玩家面对下注时不能 check，合法 call/raise/all-in 能推进状态。
- 摊牌后能显示获胜者、金额和牌型。
- 边池和分池测试通过。
- `tests/test_runner.gd` 输出 `All poker tests passed.`。
- `tests/ui_layout_probe.gd` 输出 `All UI layout probes passed.`。
- 设置、统计确认、事件日志、盲注标识和关键生成式美术在主场景中可见且不越界。

## 待确认

| 问题 | 影响 |
| --- | --- |
| 是否需要保存当前牌局、筹码或逐手历史 | 当前只保存偏好与聚合统计；扩展后会影响持久化语义和迁移策略。 |
| 是否目标发布到桌面、Steam 或 iOS | 会影响 UI 输入、窗口、安全区、导出和发布计划。 |
| AI 难度的量化目标 | 会影响 Monte Carlo 次数、起手牌策略、个性参数和性能预算。 |
| 是否需要完整规则覆盖，例如 split remainder、heads-up 盲注细节或 tournament 规则 | 会影响规则测试和产品验收范围。 |
| 最终中文像素字体、动画/VFX 和无障碍范围 | 当前已有美术主题与本地提示音，但这些范围仍会影响资源、性能和 UI 验收。 |
